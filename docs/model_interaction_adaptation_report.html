AI Art Critic - Model Development and Adaptation

This document explains how the model was built step by step, starting from a simple baseline and moving toward a stronger final version. Instead of changing everything at once, each experiment changed a small number of things so it stayed clear what helped and what did not.

For every stage, the report shows three things in plain terms: what was changed, why that change was made, and how the change improved reliability, stability, or final accuracy.

1) Development process used in practice

  1. First, a simple starting model was built so there was a clear baseline to compare with later tests.
  2. After every run, training and validation scores were checked. In some tests, an extra test split was used for a more fair check.
  3. Improvements were added step by step: better data handling first, then stronger models, then better training settings.
  4. More complex methods were only added after the basics were working well, so it stayed clear what really helped.
  5. Each run saved model checkpoints and history files, which made comparison easy and helped recover results when needed.

Overall, this was a structured process. Each change was tested, results were tracked, and decisions were based on evidence.

3) Core training pipeline

Even when the model type changed, the training flow stayed mostly the same. This made it much easier to compare one test with another.

The order was simple and repeated each time: load a pretrained model, adapt it to the style classes, choose loss and optimizer, train for several epochs, check validation results, save the best version, and test a few sample predictions.

Keeping this structure fixed avoided confusion. So when results improved, it was easier to see that the change came from a new technique, not from a completely different pipeline.

Training + validation loop pattern
def run_one_epoch(model, loader, criterion, optimizer=None):
    is_train = optimizer is not None
    model.train() if is_train else model.eval()
    total_loss = 0.0
    total_correct = 0
    total_samples = 0

    for images, labels in loader:
        images = images.to(device)
        labels = labels.to(device)
        with torch.set_grad_enabled(is_train):
            outputs = model(images)
            loss = criterion(outputs, labels)
            if is_train:
                optimizer.zero_grad()
                loss.backward()
                optimizer.step()

        preds = outputs.argmax(dim=1)
        total_loss += loss.item() * labels.size(0)
        total_correct += (preds == labels).sum().item()
        total_samples += labels.size(0)

4) Adaptation techniques that improved the system

  • Stronger model choices: the project moved from smaller models to stronger ones, so the model could learn harder style differences better.
  • Two-step training: first train the final layer, then train the full model. This made training more stable.
  • Better balance between classes: training gave extra help to underrepresented styles so they were not ignored.
  • Better generalization: stronger image variation and regularization methods were used so the model works better on new images, not only training images.
  • Better training control: learning rate control and memory-friendly settings were used to keep training stable and improve final accuracy.

The main idea was to add improvements in the right order. First make the baseline stable, then add stronger methods one by one and check if they actually help.

5) Validation, reliability, and experiment control

  1. Each run tracked basic performance numbers, and later runs tracked extra metrics for a clearer quality check.
  2. The best saved model was chosen from validation results and used for final evaluation.
  3. When possible, a separate test check was also used to confirm the model was not just memorizing validation behavior.
  4. Run history and summary files were saved so experiments could be compared later and recovered if needed.

This made the process controlled and fair: results were not only shown once, they were saved and compared across tests.

Results so far (up to test 5) show steady improvement from the baseline to the stronger pipeline. The biggest jump comes from the later experiments where model capacity and training strategy were both improved.

Top-k metric tracking
def topk_accuracies(logits, targets, topk=(1, 5)):
    max_k = min(max(topk), logits.size(1))
    _, pred = logits.topk(max_k, dim=1, largest=True, sorted=True)
    pred = pred.t()
    correct = pred.eq(targets.view(1, -1).expand_as(pred))

    results = []
    for k in topk:
        kk = min(k, logits.size(1))
        correct_k = correct[:kk].reshape(-1).float().sum(0)
        results.append(correct_k.item() / targets.size(0))
    return results

6) Integration readiness and responsible deployment checks

The model part is ready to be connected to an app. You can train it, save it, load it again later, and use it to predict the style of a new uploaded painting image.

But a real app also needs safety rules around uploads and stored data. The table below shows this in an easy way: what is already handled in the notebooks, and what must still be enforced in the backend.

Area Implemented in notebooks What still needs to be added in the app/backend
Security The prediction flow is stable: load model, process image, return result. Only allow real image files, block files that are too large, and clean file names so nobody can upload harmful files.
Privacy The model does not need personal profile information to classify art style. Do not keep user uploads longer than needed, and do not store unnecessary personal details in logs.
Ethics The style labels are clear and evaluation is tracked, so users can see what the model was trained to predict. Show confidence with predictions and clearly say when the model is unsure, especially for very similar styles.
Compliance Results, checkpoints, and experiment files are saved in an organized way. Keep clear records of which dataset and model version are used, so results can be traced later if needed.

Short version: the model is ready, but the app still needs upload rules and clear documentation. That is what makes the final deployed system safe, understandable, and reliable for real users.

7) Conclusion

The model was improved step by step instead of changing everything at once. That made it easier to see what worked and why the scores improved.

By the later tests, the project had a stronger model setup, better training techniques, and clearer tracking of results. In practice, this means the model became more accurate, more stable, and easier to use in a real app.
