AI Art Critic - Model Development & Adaptation

This document covers how I built, tested, and gradually improved the model across 12 experiments. The focus is on model interaction & adaptation: developing and validating an AI solution step by step, making sure each change actually helped before moving on to the next one.

Each experiment changed one or two things at a time so it stayed clear what made the difference. Security, privacy, ethics, and compliance are also covered at the end.

12 experiments ResNet → ViT → DINOv2 53% to 78.3% accuracy Two-stage fine-tuning Multitask learning

Overview

1. Development Process

Rather than picking the most complex approach straight away, I started simple and worked up from there. The idea was to always have a working baseline so I could tell whether a new change actually helped or made things worse.

  1. Built a simple starting model first, just to have something working that I could measure everything else against.
  2. After every run I checked training loss, validation accuracy, and in later tests also top-5 accuracy and a separate test split. Having numbers to compare made decisions easier.
  3. Improvements went in a specific order: get the data right first, then try a stronger model, then tune the training settings. Doing it in that order meant I knew what was actually causing improvements.
  4. More advanced stuff like MixUp, EMA, gradient accumulation, and multitask learning only came in once the simpler things were working well. Adding complexity too early just makes bugs harder to find.
  5. Every run saved checkpoints and a history file. That meant I could go back and compare any two experiments, and also recover results if something crashed.

Training Pipeline

4. Core Training Pipeline

Even as the model type changed between tests, the core training loop stayed the same every time. That consistency made it easy to compare results fairly: if accuracy went up, I knew it was because of the specific thing I changed, not because the whole setup was different.

The pattern was always: load pretrained weights, swap out the classification head for 27 style classes, pick a loss function and optimizer, run the training loop, save the best checkpoint.

Transfer learning setup (tests 1-2, ResNet18 baseline)

Baseline: load pretrained ResNet18 and replace the head
weights = models.ResNet18_Weights.DEFAULT
model = models.resnet18(weights=weights)

# Replace the last layer with one that outputs 27 classes (one per art style)
in_features = model.fc.in_features
model.fc = nn.Linear(in_features, num_classes)
model = model.to(device)

criterion = nn.CrossEntropyLoss()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-4)

Train/validation loop

The same loop function was used in every test. Passing an optimizer makes it train, leaving it out makes it evaluate. This avoids having two separate copies of nearly identical code.

Single function handles both training and evaluation
def run_one_epoch(model, loader, criterion, optimizer=None):
    is_train = optimizer is not None
    model.train() if is_train else model.eval()
    total_loss, total_correct, total_samples = 0.0, 0, 0

    for images, labels in loader:
        images, labels = images.to(device), labels.to(device)
        with torch.set_grad_enabled(is_train):
            outputs = model(images)
            loss = criterion(outputs, labels)
            if is_train:
                optimizer.zero_grad()
                loss.backward()
                optimizer.step()

        preds = outputs.argmax(dim=1)
        total_loss    += loss.item() * labels.size(0)
        total_correct += (preds == labels).sum().item()
        total_samples += labels.size(0)

    return total_loss / total_samples, total_correct / total_samples

Top-k accuracy tracking

From test 4 onwards I also tracked top-5 accuracy. That means: was the correct style in the model's top 5 guesses? For art style classification with 27 classes that can be quite forgiving, but it's still useful to see how close the model gets when it's wrong.

Computing top-1 and top-5 accuracy from model outputs
def topk_accuracies(logits, targets, topk=(1, 5)):
    max_k = min(max(topk), logits.size(1))
    _, pred = logits.topk(max_k, dim=1, largest=True, sorted=True)
    pred = pred.t()
    correct = pred.eq(targets.view(1, -1).expand_as(pred))

    results = []
    for k in topk:
        kk = min(k, logits.size(1))
        correct_k = correct[:kk].reshape(-1).float().sum(0)
        results.append(correct_k.item() / targets.size(0))
    return results

Adaptation Techniques

5. Two-Stage Fine-Tuning

From test 4 onwards I switched to a two-stage training approach. The idea is that a pretrained model already knows a lot about images from ImageNet training, and if you immediately update all the weights with a high learning rate you can destroy that knowledge.

So the first stage only trains the new classification head while everything else is frozen. Once the head has settled, stage two unfreezes the whole model and fine-tunes everything with a much lower learning rate.

Freeze backbone, then unfreeze for full fine-tuning
def freeze_backbone(model, model_name: str):
    # Lock all weights
    for p in model.parameters():
        p.requires_grad = False
    # Unlock only the classification head
    for p in get_classifier_parameters(model, model_name):
        p.requires_grad = True

def unfreeze_all(model):
    for p in model.parameters():
        p.requires_grad = True

# Stage 1: train head only (5 epochs)
freeze_backbone(model, cfg.model_name)
optimizer_head = torch.optim.AdamW(
    filter(lambda p: p.requires_grad, model.parameters()),
    lr=cfg.head_lr
)

# Stage 2: fine-tune everything with lower learning rates
unfreeze_all(model)
optimizer_ft = torch.optim.AdamW([
    {"params": backbone_params, "lr": cfg.ft_backbone_lr},
    {"params": head_params,     "lr": cfg.ft_head_lr},
])
The backbone and head get different learning rates in stage 2. The backbone gets 1e-5 (very small, don't overwrite what it already learned) and the head gets 3e-5 (a bit higher, it still has more to learn).

6. MixUp, CutMix, and Label Smoothing

These three techniques all work on the same problem: stopping the model from being overconfident on the training data. If you just train on raw images and hard labels, the model can end up very certain about training examples but wrong on anything slightly different. These techniques make training a bit harder on purpose so the model learns more general patterns.

  • MixUp: blends two images together and creates a blended label. For example, 60% Impressionism + 40% Baroque creates a mixed image and a target of [0.6, 0.4] instead of [1.0, 0.0].
  • CutMix: pastes a rectangle from one image onto another. Similar idea but it keeps hard edges instead of blending pixels, so the model still learns local texture details.
  • Label smoothing: instead of a "1.0" label for the correct class, it uses something like 0.92 and spreads the remaining 0.08 across other classes. This makes the model slightly less certain by design.

7. Gradient Accumulation, EMA, and Early Stopping

Three more techniques that all deal with training stability.

Gradient accumulation

The ViT model at 384px input is quite large. I couldn't fit a big batch in memory, so I used a batch size of 12 but accumulated gradients over 4 steps before updating the weights. That means the model effectively trains on 48 images at once, which is more stable than 12. The weights only update every 4 steps.

Gradient accumulation over 4 steps (effective batch = 48)
accum_steps = cfg.effective_batch_size // cfg.batch_size  # = 4

optimizer.zero_grad()
for step, (images, labels) in enumerate(train_loader):
    images, labels = images.to(device), labels.to(device)
    images, soft_targets = mixup_cutmix(images, labels, ...)

    with torch.cuda.amp.autocast():
        logits = model(images)
        loss = criterion(logits, soft_targets) / accum_steps  # scale loss

    scaler.scale(loss).backward()

    if (step + 1) % accum_steps == 0:
        scaler.unscale_(optimizer)
        torch.nn.utils.clip_grad_norm_(model.parameters(), cfg.grad_clip_norm)
        scaler.step(optimizer)
        scaler.update()
        optimizer.zero_grad()
        ema.update(model)  # update EMA weights after each real step

EMA (Exponential Moving Average)

Instead of always using the model from the most recent training step (which can be noisy), EMA keeps a smoothed version of the weights that slowly follows the training weights. At evaluation time I used the EMA weights instead. The result is usually a bit more stable and often slightly more accurate.

Early stopping

I set a patience value in each test. If the validation accuracy didn't improve for that many consecutive epochs, training stopped automatically. This prevented wasting time on epochs that weren't helping, and also stopped the model from starting to overfit.

Early stopping logic
best_val_top1 = 0.0
patience_counter = 0

for epoch in range(total_epochs):
    val_top1, val_top5 = evaluate(ema_model, val_loader)

    if val_top1 > best_val_top1:
        best_val_top1 = val_top1
        patience_counter = 0
        torch.save(ema_model.state_dict(), checkpoint_path)  # save best
    else:
        patience_counter += 1
        if patience_counter >= cfg.patience:
            print(f"Early stopping at epoch {epoch}")
            break

Test 9: Multitask Learning

8. Test 9: Training Three Tasks at Once

Test 9 is the most ambitious experiment. Instead of just predicting art style, the model now predicts style, artist, and emotion all from the same image, using the same backbone network. The idea is that learning all three things at once should push the backbone to pick up on more visual detail than it would from style labels alone.

  • Style head: 27 art style classes (Impressionism, Cubism, etc.), same as every previous test
  • Artist head: predicts which artist made the painting, extracted from the WikiArt folder structure
  • Emotion head: 7 emotion classes (anger, disgust, fear, joy, sadness, surprise, trust), from the ArtEmis dataset

Three-head architecture

Each head is a small MLP (two linear layers with a normalisation layer and activation in between) connected to the same shared ViT backbone. The backbone outputs one feature vector per image, and all three heads read from that same vector.

Building the three task heads on top of a shared backbone
class MultiTaskModel(nn.Module):
    def __init__(self, backbone, feat_dim, n_style, n_artist, n_emotion):
        super().__init__()
        self.backbone = backbone

        def make_head(out_dim):
            return nn.Sequential(
                nn.Linear(feat_dim, feat_dim),
                nn.BatchNorm1d(feat_dim),
                nn.GELU(),
                nn.Linear(feat_dim, out_dim)
            )

        self.style_head   = make_head(n_style)
        self.artist_head  = make_head(n_artist)
        self.emotion_head = make_head(n_emotion)

        # Learnable weights for each task's loss contribution
        self.log_var_style   = nn.Parameter(torch.zeros(1))
        self.log_var_artist  = nn.Parameter(torch.zeros(1))
        self.log_var_emotion = nn.Parameter(torch.zeros(1))

    def forward(self, x):
        features = self.backbone(x)
        return (
            self.style_head(features),
            self.artist_head(features),
            self.emotion_head(features),
        )

Handling missing emotion labels

Not every WikiArt image has an emotion annotation in ArtEmis. Rather than throwing those images away, I kept them and just skipped the emotion loss for that row. Emotion loss is only calculated where a valid emotion label exists.

Combine three losses with learnable uncertainty weighting
# Compute individual losses
style_loss  = ce_style(style_logits, style_labels)
artist_loss = ce_artist(artist_logits, artist_labels)

# Only compute emotion loss where annotation exists
if has_emotion.any():
    emotion_loss = ce_emotion(
        emotion_logits[has_emotion],
        emotion_labels[has_emotion]
    )
else:
    emotion_loss = torch.tensor(0.0, device=device)

# Uncertainty-weighted combination (learned during training)
loss_total = (
    0.5 * torch.exp(-model.log_var_style)   * style_loss   + 0.5 * model.log_var_style   +
    0.5 * torch.exp(-model.log_var_artist)  * artist_loss  + 0.5 * model.log_var_artist  +
    0.5 * torch.exp(-model.log_var_emotion) * emotion_loss + 0.5 * model.log_var_emotion
)

Why the 85.3% result looks misleading

Test 9 reports a style accuracy of 85.3%, which is much higher than test 8's 76.7%. But these numbers can't be fairly compared. The reason is that test 9 can only evaluate on images that exist in both WikiArt and ArtEmis, meaning only the subset that has a matching emotion label. That subset isn't random: it tends to contain the more famous, well-known paintings that are easier to classify. Rarer styles with fewer famous works are underrepresented in ArtEmis, so they show up less in the evaluation. In other words, the 85.3% is measured on an easier slice of the data, not the full validation set. It's a useful number for tracking how the multitask model is doing internally, but it doesn't mean the model suddenly got 8 points better at style classification overall.

Tests 11–12: New Backbone

9. DINOv2 and Layer-wise Learning Rate Decay (LLRD)

Tests 11 and 12 introduced a completely different backbone: DINOv2. Up to test 10, every model used a ViT trained on ImageNet with standard supervised learning. DINOv2 was trained differently: it learned by comparing images to each other (self-supervised) using 142 million images, without any human-provided labels. That extra training makes its features richer, especially for fine-grained differences like distinguishing art styles that look visually similar.

What is LLRD?

LLRD stands for Layer-wise Learning Rate Decay. A transformer model has many layers stacked on top of each other. The first layers learn basic things like edges and textures. The later layers learn more specific, high-level patterns. When fine-tuning, you don't want to change the early layers much because they already have good general knowledge. LLRD solves this by giving each layer a slightly smaller learning rate the deeper you go into the network (i.e. the earlier the layer, the lower the rate).

LLRD: earlier transformer blocks get lower learning rates
# Each transformer block gets a learning rate multiplied by decay^(depth from top)
llrd_decay = 0.75
base_lr    = cfg.ft_backbone_lr  # e.g. 1e-4

param_groups = []
for block_idx, block in enumerate(reversed(model.backbone.blocks)):
    lr = base_lr * (llrd_decay ** block_idx)
    param_groups.append({"params": block.parameters(), "lr": lr})

# The later (top) blocks train faster, early blocks train slower

Resolution: 336px → 448px

DINOv2 uses patch size 14. At 336px that gives 24×24 = 576 patches per image. Test 12 raised the input to 448px, giving 32×32 = 1024 patches, which gives the model more detail to work with. The test 12 model was warm-started from the test 11 checkpoint rather than starting from scratch, so it didn't need as many epochs to reach a good result.

Test 12 reached 78.3% val accuracy, the best result across all 12 tests, beating the previous best (test 8, ViT 76.7%) by 1.6 percentage points.

Responsible Development

11. Security, Privacy, Ethics, and Compliance

The model training side is well in hand, but if this ever turns into a real app where users upload images, there are extra things that need to be in place. Here's an honest breakdown of what's already covered in the notebooks and what would still need to be built.

Area What's already handled What still needs to be added
Security The prediction flow is clean and isolated: load model, process image, return a result. No external calls, no file writes during inference. For uploads: validate that the file is actually an image (check MIME type and file header, not just the extension), limit file size, sanitise filenames so a crafted filename can't cause path traversal issues.
Privacy The model doesn't need any personal data to work. It only looks at pixel values. Training data came from publicly available art datasets (WikiArt, ArtBench, ArtEmis). For a real app: don't store uploaded images longer than needed, make it clear to users what happens to their image, don't log identifiable details alongside uploaded content.
Ethics All 27 style labels are clearly defined and based on the WikiArt folder structure. The model's training history and evaluation numbers are fully documented. There are no hidden labels or undocumented classes. Predictions should come with a confidence score so users know when the model is unsure. Some styles are visually very similar (e.g. Impressionism vs Post-Impressionism) and the model should communicate that uncertainty rather than always committing to one answer.
Compliance Checkpoints, training configs, and accuracy history are saved for every run. Every experiment is in a numbered notebook so the full history is traceable. All datasets used have clear origins and are research-use datasets. For deployment: document which dataset version and model checkpoint is running in production. Keep a log of model updates so it's clear when the live model changed and what changed with it.

12. Conclusion

Going from 53% to 78.3% accuracy across 12 experiments wasn't one big leap. It was a series of smaller improvements that stacked up. The early tests were mostly about getting the pipeline right: proper data loading, a working training loop, saving checkpoints. Once that was stable, the focus shifted to stronger models, then better training techniques, then a completely different backbone.

The biggest single jump was between test 5 and test 6, which came mostly from training longer with proper early stopping, not from any exotic technique. That's a reminder that basic things like training duration often matter more than the fancy stuff.

The second biggest improvement came from switching to DINOv2 in tests 11 and 12. Using a self-supervised backbone pretrained on far more data, combined with a higher input resolution, pushed the best result to 78.3%, which is 1.6 points above the previous best from test 8. The multitask experiments (tests 9 and 10) showed that adding auxiliary tasks can help the model learn richer features, though the direct style accuracy numbers are harder to compare because they use different evaluation setups.
