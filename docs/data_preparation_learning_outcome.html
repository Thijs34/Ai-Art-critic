AI Art Critic - Data Preparation

This document shows how I sourced, cleaned, and prepared the data for the AI Art Critic model. The focus is on data & knowledge stewardship: taking responsibility for the quality of the data from start to finish, not just downloading it and hoping for the best.

The model's main job is recognising art styles from images, 27 styles from WikiArt like Impressionism, Cubism, and Baroque. In the later experiments I also brought in artist and emotion labels to see if training on multiple things at once helps.

Dataset Preparation

3. ArtBench: Building a Local Image Dataset

ArtBench-10 has 60,000 paintings spread across 10 art styles, roughly 6,000 per style. The images aren't stored in the repo though, so I had to write a download script to pull them all down and save them locally.

  1. First I opened the metadata CSV and checked that the three columns I needed (name, url, label) were actually there. If any were missing the script would immediately throw an error rather than silently doing the wrong thing.
  2. Cleaned up folder and file names so Windows wouldn't reject them because of special characters.
  3. Downloaded everything using 16 parallel threads at once. Each file got up to 3 attempts in case the connection dropped, and files that were already on disk got skipped so I could safely rerun the script.
  4. Saved three output files at the end: one with all the info, one in a simplified format that matches the WikiArt CSV layout, and one just for the 61 downloads that failed so I could see what went wrong.

4. ArtEmis: Linking Emotion Labels to WikiArt Images

ArtEmis is a dataset where people wrote about their emotional reactions to paintings. Each row has a painting name, an anchor painting to compare it to, and an emotion label. The catch is that ArtEmis only stores the painting names as plain text, not actual file paths. So before I could use it, I needed to figure out where each painting actually lives on disk.

  1. Loaded the ArtEmis CSV (237,998 rows).
  2. Scanned the entire WikiArt folder and catalogued all 81,444 image files. For each one, stored the lowercase filename (without extension) as a key that maps to its relative path.
  3. Went through every ArtEmis row and looked up both the painting name and anchor painting name in that map to fill in the real paths.
  4. Not everything matched up. 2,577 paintings had no match for the main column, and 2,194 had no match for the anchor. Those rows got removed.
  5. Saved the result as Contrastive_with_paths.csv. 233,475 rows made it through.
ArtEmis - building the name-to-path lookup

5. WikiArt: Preparation in the Experiment Notebooks

WikiArt is the main dataset I kept coming back to across all 12 tests. The core prep steps were the same every time, but each notebook tweaked or extended them — mainly the augmentation and how thoroughly bad images were caught.

Base preparation steps (used in every test)

  1. The script auto-finds the right CSV files by searching for keywords like "train" and "style" in filenames, so I didn't have to hardcode any paths.
  2. Built a dictionary that maps each label number back to its style name by reading the folder structure (e.g. folder impressionism/ = label 4). This got printed out before every run so I could verify it was right.
  3. Every image path got checked with Path.exists(). Rows pointing to files that no longer existed were dropped. From test 3 onwards I also tried actually opening each image to catch files that exist but are broken inside. Only 2 training rows turned out to have problems.
  4. Images got resized and normalised using ImageNet mean and std values, which work well for pretrained models. Training images also got augmented, and those augmentations got more aggressive with each test.
  5. Set num_workers=0 specifically for Windows because PyTorch's multiprocessing in the dataloader causes crashes there.
WikiArt - auto-finding CSV files & building label map
def find_split_csv(data_dir: Path, split: str) -> Path:
    csvs = sorted(data_dir.glob("*.csv"))
    preferred = [p for p in csvs if split in p.stem.lower() and "style" in p.stem.lower()]
    if preferred:
        return preferred[0]
    fallback = [p for p in csvs if split in p.stem.lower()]
    if fallback:
        return fallback[0]
    raise FileNotFoundError(f"Could not find a '{split}' CSV in {data_dir}")

def filter_existing_rows(df, image_root, split_name):

    removed = int((~exists_mask).sum())

6. Test 9: Multitask Learning (Style + Artist + Emotion)

Test 9 is where the data side of things got noticeably more complicated. Up until now every test was purely about style classification, but here I wanted the model to predict three things at once using one shared backbone:

  • Art style: 27 classes, taken from the WikiArt folder structure
  • Artist: pulled directly from the image file paths in WikiArt
  • Emotion: 7 classes (anger, disgust, fear, joy, sadness, surprise, trust) from ArtEmis

How the datasets were merged

  1. Started with the same WikiArt train/val CSVs as always (57,023 training images).
  2. Pulled the artist name from the file path of each image since it's already encoded in the folder structure.
  3. Loaded Contrastive_with_paths.csv and normalised all painting names into a (style_name, painting_name) key so they could be matched against WikiArt rows.
  4. Merged the two datasets on that key. A lot of paintings had emotion annotations from multiple different people, so I used a majority vote to pick one label per painting.
  5. Images with no emotion annotation at all weren't thrown out. I kept them but set emotion_label = -1 so the emotion loss would just be skipped for those rows during training.
Test 9 - dataset that returns all three labels
class MultiTaskDataset(Dataset):
    def __getitem__(self, idx):
        row = self.df.iloc[idx]
        image = self.load_image(row["relative_path"])
        style_label  = int(row["label"])
        artist_label = int(row["artist_label"])
        emotion_label = int(row["emotion_label"])   # -1 if unknown
        has_emotion  = emotion_label != -1
        return image, style_label, artist_label, emotion_label, has_emotion

# Loss is only applied when emotion annotation is available
if has_emotion.any():
    emotion_loss = ce_emotion(emotion_logits[has_emotion], emotion_labels[has_emotion])
    total_loss = style_loss + artist_loss + 0.7 * emotion_loss
else:
    total_loss = style_loss + artist_loss

The thinking behind multitask learning is that if you force the model to also guess artist and emotion, it has to pick up on more visual detail than it would from style labels alone. That shared knowledge should carry over and help with style prediction too.

7. Data Quality Standards Applied

Getting the data is only half the job. I also had to make sure it was actually usable before putting it anywhere near a training loop. Here's a breakdown of every check I put in place across the three datasets.

Quality check Where applied How it was done
Schema validation ArtBench Before touching anything, the script checks that name, url, and label are all present in the CSV. If one's missing it throws an error immediately rather than failing silently halfway through a download.
Missing file detection WikiArt (all tests) Every image path gets checked with Path.exists(). Rows that point to files no longer on disk get dropped. In practice only 2 training rows had to go, which is pretty clean.
Corrupt image check WikiArt test 3+ A file can exist on disk but still be broken inside. From test 3 onwards I also tried opening each image with Image.open() and calling verify() to catch those cases.
Download integrity ArtBench Each image download gets up to 3 retries before it gives up. Anything that still failed ended up in its own CSV so I could see exactly what didn't make it.
Path normalisation ArtEmis Painting names in ArtEmis can have different capitalisation or trailing spaces compared to the actual filenames. Everything gets lowercased and stripped before matching so those small differences don't cause false misses.
Label consistency WikiArt (all tests) The numeric labels in the CSV get mapped back to actual style names using the folder structure, and the full dictionary gets printed before every training run so I can double-check nothing got scrambled.
Class imbalance handling WikiArt test 4+ Some art styles have way more images than others. From test 4 onwards I calculated class weights based on how often each style appeared and used those to weight both the loss function and the data sampler, so rarer styles didn't get ignored.
Emotion aggregation ArtEmis / test 9 Many paintings in ArtEmis have emotion annotations from multiple different people. I resolved those by majority vote. Paintings with no annotations at all were kept but flagged as emotion = -1 so training could still use them for the style and artist tasks.
Dedicated artist CSV sourcing WikiArt test 10 Test 9 extracted artist names directly from the file path structure. From test 10 onwards I switched to using WikiArt's own artist_train.csv / artist_val.csv files, which are the authoritative source for artist labels and avoid any path-parsing errors.
Stratified val/test split WikiArt tests 11–12 The original validation set was split 50/50 into a proper validation set (used for early stopping) and a separate held-out test set (only used for final reporting). Stratified sampling ensured every class was represented in both halves at the same proportion.

9. Conclusion

Looking back at all the prep work, the biggest thing I took away is that data preparation is where a lot of the real work actually happens. It's easy to think you can just download a dataset and start training, but in practice there are always files missing, column names that don't match, painting names that are spelled slightly differently depending on the source, images that exist on disk but crash when you try to open them, all kinds of small things that add up.

Across this project I worked with three separate datasets (WikiArt, ArtBench, ArtEmis) that each had their own issues. ArtBench needed a full download pipeline with retry logic. ArtEmis needed a path-matching step before it could even be used. WikiArt was the most straightforward but still needed consistent cleaning and label validation across all 12 experiments.

By test 9 the data pipeline had grown quite a bit: merging two datasets, handling multi-labels, dealing with images that only have some labels available, and keeping all of that consistent across train and validation. Tests 10 through 12 refined things further switching to authoritative artist CSVs, properly splitting the validation set into a val and a held-out test set, and scaling up the input resolution, so the reported numbers are increasingly trustworthy.
